# この記事の背景

うちの会社は、エンジニアに対して少し甘すぎる気がします。

開発環境、使う技術、働き方。
細かく決められていることはほとんどなく、
「自分で考えて決める」ことが前提になっています。

この記事では、そんな環境で仕事を出来るようにどのように何をしているのか整備しているのかを、主に技術面を主として記録として書いてみます。

## 入社当時

私の会社は主に企業で利用されるソフトウェアの開発をしている若くて小さな会社です。
現在の会社は 2022 年 1 月に入社し、5 年目を迎えるこの機会に記録をしておきたい。と思い立ち書き始めました。

ひとり目のエンジニアとして入社しました。
当時は、まだまだ環境が整っておらず、社長と相談をしながら環境を徐々に整えていた事を今思えばなつかしく思います。
当時はコロナ禍ということもあり、大くの企業が主にリモートで仕事を進めていた時期でした。
我々も例外ではなくフルリモートでした。 また、今もフルリモートで仕事ができていることは感謝の念でいっぱいです。

# 開発環境

社長は根からのエンジニアで、技術的な事に対してかなり寛容です。
仕事用の PC は予算内であれば自由に選べます。
そこをケチって生産性が下る事に理解があるためです。
(限度はある気はしますが)

私はミーティングや事務作業で利用するノート PC と、開発用の PC の2台を使い仕事を進めています。

## 私の PC

### ノート PC

現在は ThinkPad X1 Carbon Gen 13 を使っています。
(用途的にオーバースペックだが、お願いをしたタイミングが良かったのかリクエストより高スペックだった)

OS は ArchLinux を入れ、 Desktop Environment は Gnome です。
今年、仕事環境をフル Linux にできたため、今年が Linux Desktop 元年です。私にとっては。

もちろん、職場で Linux を強制している分けではなく、ほとんどの方が Windows 、たまに Mac となっています。
Linux 利用は、私含め数名です。
(さみしい。仲間がほしい...)

Excel ファイルなどは Windows では無いため扱いが少し面倒な事もあります。
この PC の主な利用は

- WEB ミーティング
- Slack での仲間とのコミュニケーション
- SSH で開発機にアクセス

のため、それ以外は困ること無く生活できています。

### ヘッドレスの開発機

開発機は EPSON Endeavor SG 150 を使っています。
メモリはそこそこあり、データ領域は Btrfs で RAID1 を組んでます。
もちろん Arch Linux を入れていて、ディスプレイ無しのヘッドレス運用をしてます。

なお、 Arch Linux は入社以来、開発環境として利用していますが、目立ったトラブルは無いです。

## ソフトウェア

### TailScale

普段はノート PC と開発機は自宅にあるため、ネットワークで困ることは無いのですが、問題は数ヶ月に一度程度ある出社の時です。
そこは、自社で Tailscale を導入しており、開発機は自宅。出先でノート PC から SSH 接続で普段通りに開発できます。

Tailscale は中央サーバーの無い P2P を基本とした VPN です。アカウントを登録しており、通信を許可していれば機器間での通信が出きるものです。もちろん人だけではなく、サービスも同様です。

### エディタ

自社は開発環境も特に決まりはありません。

もちろん、何をやっても良いわけではありません。
ただ、ルールで縛る代わりに、
説明できること・責任を持てることが求められています。

自由である分、考える量は多い環境だと思います。
ただ、全員に多くを考えさせることは本意ではないので、ある程度自由に出来る余地を残してレールを引くことが大事だと思います。そのレールを敷く役割が今のところ私だ。というところです。

自身でトラブル対応ができないのであれば、 Visual Studio Code で開発することを推奨しています。

ということで、私自身は Neovim を使っています。
個人用に使っている dotfiles をリポジトリ管理しているため、
個人的な開発と仕事での環境は、同じ手に馴染んだ環境です。

この全員がバラバラな前提の環境で、開発手順の整備が大変では？となると思いますが、そこは Dev Containers です。

### Dev Containers

ここまで記事を読んで下さっている方であれば、 Dev Containers はご存知かと思いますので詳細は他の優良な記事に譲ります。
(分からなかったらすみません。言い分けです。)

開発案件開始直後に、私がリポジトリに案件に特化した Dev Containers をつっこんで、「このルールに従ってね」というスタンスでやっております。
これ、開発手順書を書かなくてよいので、とても良く感じています。ありがとう。 Visual Studio Code。

私は Neovim なので、 [devcontainer-cli](https://github.com/devcontainers/cli) を直に使っています。
(LSP は手元、実行だけコンテナ内で少しいびつな運用をしていますが、結局これが一番上手く動く。改善はしたいが。)

### 1Password

`.env` に格納が必要な認証情報の管理は大変ですよね。
共有するにも、平文で Slack に投稿できない。Wiki にも可能であれば載せたくない。

自社では、 1Password を利用しており、 `.env` などの共有をしています。
先述の Dev Containers 環境内からであれば、手間をかけずに取得できるように整備しています。

## 開発物の管理

コード管理は GitHub を使い、特に縛りが無ければ、インフラは AWS を利用しています。
→ 基本、開発は請け負いで頂けているため自由が効く場合が多いです

開発言語は、特に縛りがあるのでは無いが、 TypeScript が主になっています。
(個人的には Go あたりが仕事にマッチしていると考えてますが、手が足りず布教できてません。)

AWS は、基本的に CDK で記述しているため、 CI / CD で、

- 開発ブランチはマージで開発環境にデプロイ
- 本番系はタグを打つことでリリース

と、基本自動化されています。

# さいごに

もともとは、 Neovim のプラグインが...や、使っているターミナルは... などのもっと技術よりの話を書きたいと思っていました。が、想像以上に量が多くなりそうだったため、省きました。
(かなりポエムになってしまった。)
自由にできる環境を整えた上で、私自身は自由にさせてもらっています。

以前の会社は、いわゆる伝統的な SI 系の企業で、全く自由の効かない企業でした。
その反動からか、今の会社に声をかけて頂き入社してからは自由の効く環境にできるよう整備してきました。
まだまだ改善すべき所は多くあり、終りは見えません。
ただ、これからも DX (Developers Experience) 改善に努め、皆が楽しく、クリエイティブに開発ができるよう、活動を進めていきたいと思います。

このやり方が正解だとは思っていません。
ただ、こういう仕事の仕方もある、という話です。

今の環境に少し息苦しさを感じている人が、
外を見るきっかけになれば嬉しいです。

また、そのように出来るよう私以外の人も努力されている結果、このように仕事ができていることを感謝しています。
(特に社長は、このように仕事が出来るよう、並々ならぬ苦労をなさっているかと)

宣伝みたいになることが嫌なので、会社名は出しませんでした。
もし、このような職場へ興味を持っていただけるのであれば、私の X や GitHub に連絡先があるのでぜひ！
フロントエンド、バックエンド、インフラ、スクラッチ開発でなければ Salesforce など色々なことが出きます。
(Linux 仲間が少ないので、さみしい)
